## 批改总结：不足与待提升

---

### 1. 推导式思维没有内化

练习 3 的简单场景能用推导式，但练习 4 一复杂就退回了 for 循环 + 手动维护变量的命令式写法。`highest`、`passed`、`class_avg` 三个值都可以用推导式 + `max()` / `sum()` 一行搞定，你却用了一个 for 循环加 if 判断逐个累加。**"从列表生成列表"用推导式、"从列表求聚合值"用 `sum()/max()/min()` + 推导式**——这个条件反射还没建立。

### 2. `[]` 和 `.get()` 不区分场景

所有 dict 访问一律用 `.get()`，包括数据结构已知、key 一定存在的场景。这会让代码表达不出意图——别人读你代码时分不清这个 key 到底是"可能不存在所以防御"还是"一定存在"。已知结构用 `[]`，不确定时用 `.get()`。

### 3. `isdigit()` 做类型转换判断不可靠

负数、小数、带空格的数字字符串都会判断失败。这题数据恰好是正整数所以没暴露问题，但这是一个会在真实项目中埋雷的习惯。类型转换的标准做法是 `try/except`，你在 Day 2 练习 1 已经用过，但练习 4 里没有复用这个模式。

### 4. `setdefault()` 返回值没有利用

练习 3 第 5 题分组写了两行，`setdefault()` 返回的就是那个 value，可以直接 `.append()`。`d.setdefault(k, []).append(v)` 是 Python 分组归集的标准一行写法，要记住。

### 5. JS 语法习惯还没清除

- `if` 条件加括号（多处出现）
- 变量命名偶尔用 `camelCase` 或无描述性的 `step1_arr`
- 修改字典单个字段用 `|` 合并整个字典，而不是直接 `dict["key"]["field"] = val`

这些不影响运行但不是 Python 风格，面试和协作中会被注意到。

### 6. 防御性编程不够一致

练习 5 的 `node.get("children")` 如果 key 不存在返回 `None`，`for c in None` 会报错。需要写成 `node.get("children", [])`。你在有些地方有防御意识（用 `.get()`），但在需要给默认值的地方又漏了。

你用 for 循环手动拼字符串再 `strip()`，这正是前面说的"推导式思维没内化"的表现。标准写法：

```python
meta_str = " ".join(f"{k}={v}" for k, v in metadata.items())
```

一行替代四行，而且不需要处理尾部空格。

### 练习 4 有两个问题

**1. 去重用字符串拼接而不是 tuple，会产生歧义：**

```python
# 你的写法
f"{l['user']}{l['page']}{l['action']}"

# "Alice" + "/home" + "view" → "Alice/homeview"
# 如果有个用户叫 "Alice/hom"，page 是 "eview"，拼出来也是 "Alice/homeview"
# 虽然现实中不太可能，但这是一个不严谨的做法
```

题目提示里说了用 tuple，tuple 作为 set 元素天然没有歧义：

```python
unique = {(l["user"], l["page"], l["action"]) for l in logs}
```

**2. `exclusive_pages` 用 `^` 对称差集是错的。**

`^` 是"只在其中一个集合里的元素"，但多个集合连续 `^` 的结果不是"只有一个用户访问过的页面"。比如一个页面被 3 个用户访问，`^` 两次之后它又回来了（异或的特性）。

正确思路是统计每个页面的 UV，UV 等于 1 的就是 exclusive：

```python
exclusive_pages = {page for page, uv in page_uv.items() if uv == 1}
```

你已经算出了 `page_uv`，直接复用就行。

同理 `common_pages` 硬编码了四个用户名，如果用户列表变了就得改代码。应该动态处理：

```python
all_page_sets = list(user_pages.values())
common_pages = all_page_sets[0].intersection(*all_page_sets[1:])
```

`intersection()` 可以接收多个参数，`*all_page_sets[1:]` 把剩余的 set 展开传入——正好是今天学的 `*` 展开语法。

---

### 练习 5 思路正确，有三个问题

**1. 修改了原始数据。**

`result_configs = configs[0]` 没有复制，直接引用了 `default_config`。函数执行完之后，`default_config` 的内容被改了。这在前端里也是常见的 bug——直接修改 props 或传入的对象。应该深拷贝：

```python
import copy
result_configs = copy.deepcopy(configs[0])
```

**2. `if (old_value):` 判断有 bug。**

如果 `old_value` 是 `False`、`0`、空字符串，都会被当成"不存在"走 else 分支直接覆盖。`debug: False` 就会命中这个问题。应该判断 key 是否存在，而不是值是否 truthy：

```python
# ❌ 你的写法 — debug=False 会被当成不存在
if old_value:

# ✅ 判断 key 是否存在
if key in result_configs:
```

这是 Python 和 JS 共有的经典 falsy 陷阱，值得记住。

**3. `if not index == 0` 跳过第一个的写法不够清晰。**

更 Pythonic 的做法是用切片直接从第二个开始遍历：

```python
result = copy.deepcopy(configs[0])
for config in configs[1:]:       # 从第二个开始
    for key, new_value in config.items():
        ...
```

不需要 `enumerate`，不需要 `if` 判断，代码意图更清晰。

### 练习 1 ✅ 逻辑正确，一个问题

`employees.sort()` 是原地排序，虽然你用推导式创建了新列表所以没有修改原数据，但变量名跟外部的 `employees` 同名，函数内部会遮蔽（shadow）外部变量。改个名字比如 `filtered` 更清晰，也避免潜在混淆。

进阶部分"薪资相同时按工龄降序"你注释了一行 `lambda e: (e[f"{b}"] for b in e[by])`，这个写法不对。Python 多字段排序的标准做法是 `key` 返回 tuple：

```python
# 先按 salary 降序，salary 相同再按 years 降序
employees.sort(key=lambda e: (e["salary"], e["years"]), reverse=True)
```

tuple 比较是**逐元素比较**的，第一个元素相同就比第二个，跟 SQL 的 `ORDER BY salary, years` 一样。

### 练习 2 ❌ 没有完成

`create_filter` 只写了外壳，内部的 `filter_func` 没有实现过滤逻辑，也没有 `return filter_func`。这是一个闭包（closure）的典型应用——外层函数接收条件，返回内层函数，内层函数捕获外层的 `conditions` 变量。

提示一下思路：

```python
def create_filter(**conditions):
    def filter_func(items: list[dict]):
        result = items
        for key, value in conditions.items():
            if key.endswith("_min"):
                field = key[:-4]       # 去掉 "_min" 后缀
                result = [item for item in result if item[field] >= value]
            elif key.endswith("_max"):
                # ...同理
            elif key.endswith("_contains"):
                # ...同理
            else:
                # 精确匹配
                result = [item for item in result if item[key] == value]
        return result
    return filter_func    # 别忘了返回内层函数
```

关键知识点是 `key.endswith("_min")` 判断后缀 + `key[:-4]` 切片去掉后缀提取真实字段名。试着自己补完。

### 练习 3 ✅ 整体不错，两个问题

**1. `aggregate` 里 `avg` 返回了格式化字符串而不是数字：**

```python
"avg": lambda arr: f"{(sum(arr) / len(arr)):.2f}",
```

这返回的是 `"5332.33"`（字符串），不是 `5332.33`（float）。后面 `top_n` 或任何需要数值比较的地方都会出问题。格式化是展示层的事，计算层应该返回数字：

```python
"avg": lambda arr: round(sum(arr) / len(arr), 2),
```

**2. `count` 的实现有隐含问题：**

```python
"count": len
```

你传给 `len` 的是 `[record[field] for record in group]`，也就是先提取了某个字段再数长度。对于 `count` 来说，提取哪个字段不重要，结果都一样——但这意味着如果某条记录缺少那个字段就会 `KeyError`。更准确的做法是 `count` 直接数分组的记录数，不依赖字段。不过当前数据结构完整所以没出问题，知道这个隐患就行。
