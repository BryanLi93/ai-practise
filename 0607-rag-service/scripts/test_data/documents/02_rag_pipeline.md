# RAG Pipeline Lab Notes

这份文档描述一个检索增强生成服务的主要步骤，适合测试“分块、embedding、top-k、reranker、prompt 和评估”这些问题。它与 FastAPI 文档的区别是：这里关注知识库质量和检索链路，而不是 HTTP 路由或数据库会话。一个健康的测试集应该让这些主题能被不同问题分别召回。

## 文档清洗与解析

RAG 的第一步不是 embedding，而是把原始资料整理成适合检索的文本。Markdown 可以保留标题层级，日志可以保留时间和模块名，网页需要去掉导航、页脚和广告噪声。清洗时要保留对回答有用的上下文，例如章节标题、表格标题、代码块说明和来源信息。如果只把一小段泛泛介绍塞进知识库，后续向量检索没有足够信号区分问题。

## Chunk Size 与 Chunk Overlap

chunk_size 决定每个片段能承载多少语义。太小会把完整说明切碎，导致答案缺少前因后果；太大又会把多个主题塞在一起，使相似度分数变得模糊。chunk_overlap 用于保留边界附近的上下文，适合处理长段落、步骤列表和跨段解释。测试时可以准备多个章节，每个章节围绕一个清晰问题，这样 top-k 返回结果更容易判断是否合理。

## Embedding 的文档向量与查询向量

入库时使用 RETRIEVAL_DOCUMENT 类型，查询时使用 RETRIEVAL_QUERY 类型，是很多 embedding 服务推荐的做法。文档向量和查询向量必须来自兼容模型，并且维度要和数据库向量列一致。更换 embedding 模型后，旧向量通常需要重新生成，否则相似度空间不一致。检索效果差时，不要只怀疑数据库，也要检查文本清洗、语言混合、专业术语和问题写法。

## Top-k、阈值与 Reranker

top-k 控制初步召回数量。k 太小可能漏掉真正相关的 chunk，k 太大则会把无关文本带进 prompt。向量相似度适合快速召回，但不一定能精确排序，尤其是多个 chunk 都包含相似关键词时。reranker 会把问题和候选 chunk 成对比较，重新排序后再取前几条。只有当知识库变大、主题相近、误召回明显时，reranker 的价值才会变得突出。

## Prompt 组装与回答约束

检索到的 chunks 通常会用分隔符拼接成 context，再和用户问题一起交给 LLM。prompt 应该要求模型只基于 context 回答，并在找不到依据时明确说明不知道。否则模型可能凭训练知识编造答案。练习阶段可以先打印 context，不急着生成最终答案，因为只要能确认问题召回了正确 chunk，就已经验证了 RAG 链路的关键部分。

## 评估用问题集

评估问题应该覆盖事实型、步骤型、对比型和无答案型。事实型问题检查某个定义是否能被召回；步骤型问题检查流程是否完整；对比型问题检查相近概念是否混淆；无答案型问题检查系统是否会把知识库外的问题硬答出来。每个问题最好标注预期主题或预期文档，便于人工看日志时快速判断检索是否跑偏。
